## Prompt 7 (v2): Final README — Compare Six Analysis Methods and Document What Was Learned

> **Context for this prompt:** You are being given the file `full_conversation.md` (~278 KB, ~7,043 lines) from the repository `R00S/claude-experiment`. This is a raw export of a GitHub Copilot Chat session between @R00S and Copilot. You are also being given six analysis files produced by earlier prompts: `analysis/iteration-1a.md`, `analysis/iteration-1b.md`, `analysis/iteration-2a.md`, `analysis/iteration-2b.md`, `analysis/iteration-3a.md`, `analysis/iteration-3b.md`. Each was generated by a separate AI session with no awareness of the others.

> The six iterations use three different reading methods at increasing depth, with two independent runs per method:
> - **Method 1** (iterations 1a, 1b): Single sequential pass — read the whole file and answer directly
> - **Method 2** (iterations 2a, 2b): Chunked — divide the file into ~7 logical sections, analyze each independently, then synthesize
> - **Method 3** (iterations 3a, 3b): Multi-phase structural mapping — Phase 1: build a table of contents with line numbers; Phase 2: deep per-section analysis; Phase 3: synthesis
>
> The "b" variant of each method was given an additional "honesty prompt" — an instruction to self-assess reading depth, flag areas of uncertainty, and distinguish what was actually read from what was guessed or pattern-matched. The "a" variants received no such instruction.

### THE TWO QUESTIONS

**Question 1: Does the reading method matter?**

**Question 2: Does the honesty prompt matter?**

Did asking the AI to self-assess its reading depth improve accuracy, have no effect, or make things worse? Were the self-assessments themselves accurate? Did the honesty prompt catch errors that would otherwise have gone unnoticed?

**These two questions must be the organizing principle of the entire README.** They should appear early, be answered clearly with evidence, and the reader should be able to find the verdict within the first few paragraphs of each section. Everything else — the project summary, the technology list, the method documentation — is supporting context for these two questions.

### HOW TO ANSWER THESE QUESTIONS

#### Step 1: Find the disagreements

Compare the six iterations against each other to identify every point where they **disagree** — different root causes for the same problem, different counts of problems found, different technology versions, different claims about what was working at the end of the conversation, different quotes attributed to the participants.

This is the **only valid use of cross-iteration comparison**: finding where the methods diverged. Agreement between iterations tells you nothing about correctness — see below.

#### Step 2: For every disagreement, check the source

For each disagreement found in Step 1, go to `full_conversation.md` and determine which version (if any) is supported by the actual source text. Quote or cite the relevant passage with approximate line numbers.

Possible outcomes for each disagreement:
- **One iteration is correct, others are wrong** — state which and cite the source passage
- **Multiple iterations are wrong in different ways** — state what each claimed and what the source actually says
- **The source is ambiguous or incomplete** — say so explicitly; do not pick a winner
- **The source does not address the point at all** — mark as "[not in source]"

#### Step 3: Assess accuracy by method

Using the results from Step 2, build an accuracy profile for each method:
- How many disagreements did this method get right?
- How many did it get wrong?
- How many did it fabricate (claim something specific that doesn't appear in the source at all)?
- What *kinds* of claims did it get wrong? (high-level narrative vs. specific technical details vs. exact quotes)

#### Step 4: Answer the two questions

Use the accuracy profiles to answer Questions 1 and 2 with evidence.

### CRITICAL CONSTRAINT: What comparison can and cannot tell you

**Comparison between iterations can ONLY be used to find disagreements.** It cannot be used to determine which side of a disagreement is correct.

This is not a stylistic preference. It is a structural constraint of this experiment, and here is why:

All six iterations were produced by the same LLM architecture. They share the same training data, the same reasoning patterns, and the same failure modes. When the same model runs multiple times on the same input:

- If 5 out of 6 iterations agree on a claim, that does **not** mean the claim is likely correct. It means the model has a strong tendency to produce that claim. The tendency could be toward truth or toward a plausible-sounding confabulation — you cannot tell from the agreement alone.

- If iteration 1a was produced first and iteration 3b last, that does **not** mean 1a's version is more likely correct because it came first, or that 3b's version is more likely correct because it used a more sophisticated method. Each must be checked against the source independently.

- If one iteration makes a unique claim that no other iteration makes, that does **not** mean the claim is wrong. It could be the only iteration that noticed something the others missed. Or it could be a fabrication. You cannot tell from the uniqueness alone. You can only tell by checking the source.

**The only arbiter of factual correctness is `full_conversation.md`.** Cross-iteration comparison tells you *where to look*. The source text tells you *what is true*.

This was empirically demonstrated during the cross-referencing phase of this experiment. A claim about the root cause of the `get_ha_logs` 404 error was accepted as "correct" because most iterations agreed on a particular explanation. When that claim was verified against the actual source text:
- The user proved the HA token was valid (`curl` returned `{"message":"API running."}`)
- The 404 persisted despite a valid token
- The root cause was **never definitively resolved** in the conversation
- Both "token mismatch" and "deprecated endpoint" were inferences made by the iterations, not confirmed facts from the source

The majority-voted answer was wrong. The source text was the only thing that could reveal this.

### SPECIFIC KNOWN FABRICATIONS

The cross-referencing phase identified these confirmed fabrications. Do not propagate them, and use them as calibration examples when assessing iteration accuracy:

1. **iteration-3b** lists the `run_tests` missing action types as `wait_for_state`, `fire_event`, `check_attribute`, `call_api`, `assert_log`. The actual test output in `full_conversation.md` (~line 6013) shows the real missing types were: `check_integration_loaded`, `init_config_flow`, `check_options_flow`, `import_module`, `check_logs`. The iteration-3b list is entirely fabricated — plausible-sounding test action names that don't appear anywhere in the source.

These fabrications are useful data points: they show that a more complex method (Method 3) with a honesty prompt ("b") produced fabrications that the simpler methods did not. That is directly relevant to answering both Question 1 and Question 2.
